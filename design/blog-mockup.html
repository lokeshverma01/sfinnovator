"Too many SOQL queries: 101" — the real root cause and fix

🗓 Published Jun 20, 2026 🔄 Updated Jun 21, 2026 ⏱ 6 min read 📈 Intermediate
⚠ Symptom
System.LimitException: Too many SOQL queries: 101 — thrown when saving Accounts in bulk.
🧩 Environment
Apex trigger on Account, bulk DML (Data Loader, 200 rows).
🔍 Root Cause
A SOQL query placed inside a for loop runs once per record.

What the error actually means

Salesforce enforces a governor limit of 100 SOQL queries per synchronous transaction. The 101st query throws immediately. In bulk operations this surfaces fast because the trigger runs against many records at once.

Reproducing it

  1. Create a trigger that queries Contacts for each Account in a loop.
  2. Load 200 Accounts via Data Loader.
  3. Observe the LimitException at record ~101.
// ❌ Anti-pattern: SOQL inside the loop
for (Account a : Trigger.new) {
  List<Contact> cs = [SELECT Id FROM Contact WHERE AccountId = :a.Id];
}

The fix — bulkify

Query once outside the loop and build a map in memory:

// ✅ One query, mapped in memory
Map<Id, Account> byId = new Map<Id, Account>(Trigger.new);
List<Contact> cs = [SELECT Id, AccountId FROM Contact
                    WHERE AccountId IN :byId.keySet()];
✅ Resolution
Move SOQL out of loops; query once with an IN clause and use Maps. The transaction now runs a single query regardless of volume.
🔑 Key takeaways
  • Never put SOQL/DML inside a loop.
  • Think in collections: query once, process in memory.
  • Test triggers with 200+ records to catch limits early.
